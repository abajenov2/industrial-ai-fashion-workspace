# Resident Journey and Digital Trace

Use this reference when a task involves onboarding, registration, resident journey, meetings, follow-up, publications, validation, certification, or forum workflows.

## Resident Journey Principle

Do not describe only the ideal funnel. Record micro-actions:

```text
person sees
-> person clicks / writes / reads / attends
-> system responds
-> curator helps
-> digital trace appears
-> next step becomes clear
```

For each step capture:

- screen / place;
- visible text or button;
- resident action;
- system response;
- curator role;
- risk of losing the person;
- improvement idea;
- digital trace;
- next step.

## Digital Trace Objects

Digital trace can include:

- profile and intro;
- post;
- comment;
- project;
- event participation;
- meeting transcript;
- interview;
- protocol;
- expert validation;
- certification / qualification result;
- offer or special condition;
- link to KA / DocKA / TR / TRxTR / RLTR / A1.x.

Use the workspace indexing rule:

```text
TR = digital trace of one resident / organization
TRxTR = concrete case where several trajectories intersect
A1.x = repeatable modular assembly derived from a requirements matrix, typical modules, and several similar TRxTR cases
RLTR = reference role trajectory derived from multiple TR/TRxTR
```

When extracting `RLTR` knowledge from a meeting, keep the levels separate:

```text
one person's or organization's movement -> TR
the moment another role enters that movement -> TRxTR
the repeatable transition supported by several TR/TRxTR -> RLTR candidate
validated and practically tested role route -> reference RLTR
```

Name the `RLTR` after the role. A person may provide the strongest initial
evidence but does not become the trajectory itself. For every `TRxTR`, record
why the other role became necessary exactly then, what prior evidence created
trust, what small product or action opened the work, and whether the outcome is
already verified. Do not describe a started pilot as a proven case.

Do not label one meeting, pilot, participant, direction, or case as `A1.x`. Label it as `TRxTR`; propose `A1.x` only after similar cases repeat and a requirements matrix plus typical modules can be described.

Working formula:

```text
requirements matrix + library of typical modules + unique assembly for a contour -> A1.x
```

Approved `TRxTR` artifacts usually belong in the Alliance room `AI оптимизация`. This room is the laboratory layer for digital trace, workspace setup, validation, resident support, A1.x testing, and Industrial AI learning. Sensitive cases stay private, anonymized, or closed until approved.

## Meeting Processing

```text
calendar event
-> meeting / recording
-> raw transcript
-> processed artifact
-> follow-up
-> site update / task / TR / TRxTR / KA / DocKA / A1.x
```

Choose one primary artifact:

- `TR`: movement of a resident, project, or organization;
- `TRxTR`: concrete case where several residents, roles, or organizations intersect;
- `A1.x`: repeatable modular assembly that has needs, requirements, typical modules, validation logic, and repeatability;
- `interview`: voice, position, experience of a person;
- `meeting protocol`: decisions, tasks, attendees, follow-up;
- `Knowledge Base draft`: generalized method or standard.

## Validation / Certification

Validation should be based on digital trace, not only self-description.

Look for:

- role;
- stage of maturity;
- evidence;
- expert comment;
- next recommendation;
- access to cooperative chain;
- relation to TRxTR / RLTR / A1.x.
