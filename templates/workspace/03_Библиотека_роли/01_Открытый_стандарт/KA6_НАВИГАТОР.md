# KA6: навигатор рабочего места

KA6 помогает не читать весь Свод подряд, а определить свою точку и построить
проверяемый маршрут.

## 1. Где я сейчас

- моя роль или совмещение ролей `RL`;
- стадия компании / практики: `XS / S / M / L` или `needs_validation`;
- текущая реальная траектория `TR`;
- ближайшая проблема, решение или возможность;
- ограничения по времени, деньгам, данным и доступу.

Если данных не хватает, Codex не придумывает роль или стадию, а помечает их
`needs_validation`.

## 2. Куда я двигаюсь

- желаемое состояние и наблюдаемый результат;
- эталонная ролевая траектория `RLTR`;
- следующий уровень зрелости;
- срок ближайшего проверочного действия.

`KA6` находит и валидирует маршрут. `RLTR` описывает содержание эталонной
траектории роли. Личный путь остается в `TR`, а совместный кейс — в `TRxTR`.

## 3. Какие знания нужны

1. Найти главный дом вопроса в `KA1-KA5`.
2. Прочитать вводный узел и релевантные статьи.
3. Проверить связанные роли, экспертов, кейсы и TRxTR.
4. Выбрать только нужный документ DocKA.
5. Зафиксировать, что именно будет проверено на практике.

## 4. Когда подключается другая роль

KA6 должен показывать не просто список экспертов, а точку на RLTR, где другой роли становится нужна проверенная помощь.

```text
этап моей RLTR
+ текущее ограничение
+ цена ошибки
+ доказанный кейс другой роли
+ небольшой проверяемый продукт
-> подходящий TRxTR
-> действие + владелец + дата
```

Пример: эксперт по ассортименту и экономике канала подключается к марке, когда она начинает новый канал без собственного опыта или уже видит первые отрицательные показатели. Конкретная совместная работа фиксируется как `TRxTR`; после результата она уточняет обе RLTR.

KA6 передает доверие не от абстрактного рейтинга экспертов. Сначала человек должен увидеть кейс похожего участника и узнать собственную ситуацию:

```text
похожий человек или бизнес
-> узнаваемая точка А и момент, когда помощь стала полезна
-> доказанный переход клиента
-> эксперт, который помог
-> ограниченный первый такт
```

Так сработала связка Knit&Fit -> Miditi: Алина Цуцу отправила Дарье Титовой ссылку на кейс похожей марки, Дарья узнала собственную ситуацию с выходом на Lamoda и пришла к Алине. Это подтверждает правило «люди идут за людьми»: доверие возникает через опыт таких же участников, а не только через самоописание эксперта.

Проверенное правило навигации для маркетплейс-канала:

```text
марка выходит в новый маркетплейс-канал
+ нет внутренней компетенции по unit-экономике
+ не учтены хранение / возвраты / логистика / продвижение
+ есть риск отрицательной рентабельности
-> показать доказанный кейс Knit&Fit x Алина Цуцу
-> предложить эксперта с подтвержденным похожим переходом клиента
-> начать с ограниченного расчетно-аналитического пилота
-> зафиксировать внедрение, повторный замер и новые правила управления каналом
```

Источник доказательства: [публичная фиксация Knit&Fit x Алина Цуцу](https://alliance.beinopen.ru/post/3795/#Keis-Iany-i-Aliny-Rabota). Рекомендация строится по доказанному переходу клиента, а не по самоописанию эксперта.

## 5. Ладошка Сачек для модели компании

Проверить согласованность пяти верхнеуровневых документов:

| KA | Документ |
|---|---|
| KA1 | стратегия компании |
| KA2 | бренд-стратегия |
| KA3 | маркетинговая стратегия |
| KA4 | ассортиментная стратегия |
| KA5 | коммерческая стратегия |

Начинать нужно с отсутствующего или противоречивого документа, который сильнее
всего блокирует ближайшее действие, а не с механического заполнения всех пяти.

## 6. Результат навигации

```text
роль + стадия + TR + запрос
-> главный KA
-> статья / кейс / эксперт / RLTR
-> выбранный DocKA при необходимости
-> действие + владелец + дата
-> цифровой след результата
-> уточнение траектории
```

- текущая точка:
- целевая точка:
- главный KA:
- связанная RLTR:
- выбранный материал / DocKA:
- следующий шаг:
- владелец:
- дата:
