# Стандарт развития библиотеки Альянса

Этот файл задает единый формат для новых CJM, ролей, траекторий, правил и
повторяемых сборок. Он нужен, чтобы человек мог читать библиотеку как учебник,
а Codex мог искать по ней без догадок.

## Два представления одного объекта

Каждый утвержденный объект имеет:

1. Markdown-карточку с объяснением, контекстом и пошаговой логикой.
2. YAML или JSON-запись со стабильными полями для поиска и связей.

Markdown является читаемым содержанием. Машинная запись не должна добавлять
смысл, которого нет в утвержденной карточке или источнике.

## Минимальные поля

```yaml
id: stable-id
title: Читаемое название
artifact_type: cjm | role | rltr | assembly | rule | protocol
version: 0.1
status: draft | preview | approved | archived
access: public | resident | private | pending_rights
owner_role: роль владельца стандарта
source_urls: []
related_ka: []
related_docka: []
related_roles: []
related_journeys: []
related_assemblies: []
updated_at: YYYY-MM-DD
```

Для CJM дополнительно нужны входное и выходное состояние, шаги, контрольные
точки и критерий завершения. Для роли нужны RBS, FBS, SBS, WBS и связи DSM. Для
сборки нужны матрица требований, модули, роли, документы, ограничения и
доказательство повторяемости.

## Стабильные идентификаторы

- `CJM-RESIDENT-...` - путь резидента;
- `RL-...` - роль;
- `RLTR-...` - эталонная ролевая траектория;
- `A1.x` - проверенная повторяемая сборка;
- `RULE-...` - правило;
- `PROTOCOL-...` - протокол.

ID не меняется при редактуре названия. Смена смысла требует новой версии или
нового объекта.

## Порядок добавления

```text
реальная задача или повторяющийся вопрос
-> черновик Markdown
-> ссылка на источник и проверка доступа
-> машинная карточка
-> проверка связей с KA, DocKA, ролями и кейсами
-> проверка владельцем
-> утверждение роли Архитектора
-> новая версия GitHub-пакета
```

Не превращайте единичный кейс в общий CJM или A1.x без повторяемости. Сначала он
фиксируется как TR или TRxTR.

## Что хранить только ссылкой

- актуальные профили и контакты;
- события и комментарии;
- резидентские статьи без разрешения на открытое зеркало;
- DocKA и другие файлы с ограниченным доступом.

Ссылка должна вести на источник Альянса, а карточка должна объяснять, зачем к
нему переходить и какое право доступа требуется.

## Контроль качества

Перед выпуском проверяются:

- уникальность ID;
- существование всех локальных файлов;
- доступность внешних ссылок по возможности;
- отсутствие личных данных и закрытых тел в публичном пакете;
- соответствие Markdown и машинной карточки;
- наличие следующего действия с датой в операционных CJM;
- отсутствие выдуманных ролей, полномочий, авторства и связей.
